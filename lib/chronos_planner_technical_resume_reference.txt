## ARCHITECTURE & DESIGN PATTERNS

### Layered Repository Architecture
```
UI Layer (Screens/Widgets)
    ↓ (depends on abstractions)
Provider Layer (ChangeNotifiers)
    ↓ (depends on interfaces)
Repository Interfaces (Abstract Contracts)
    ↓ (implemented by)
Local Repository Implementations (Drift-backed)
    ↓ (uses)
Data Access Objects (DAOs - Drift Generated)
    ↓ (queries)
SQLite Database (Local Persistence)
```

### Key Architectural Patterns
- **Repository Pattern** - Abstract contracts (`ScheduleRepository`, `TemplateRepository`, `PreferenceRepository`, `TodoRepository`) enabling future backend swapping (e.g., cloud sync) without UI changes
- **MVVM Principles** - Clear separation: UI (Views) ↔ Providers (ViewModels) ↔ Repositories (Models)
- **Dependency Injection** - MultiProvider at app root, repository instances injected into providers
- **Optimistic UI Updates** - In-memory state updated immediately, async persistence follows
- **Reactive Streams** - Drift `.watch()` methods for real-time UI updates without manual refresh
- **Undo/Redo System** - Stack-based undo with task snapshot tracking (`_UndoAction`)
- **Offline-First Design** - Zero network dependencies, fully functional offline application

## DATA MODELS & DATABASE SCHEMA

### Database: SQLite via Drift ORM (Schema Version 5)
**Database File:** `chronos_planner.sqlite` (platform-specific app documents directory)

### Tables & Relationships
| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `Tasks` | Individual scheduled tasks | id (PK), title, description, startTime, endTime, type, priority, energyLevel, estimatedCost, actualCost, completed, dayPlanId (FK→DayPlans), sourceTemplateId (FK→PlanTemplates) |
| `DayPlans` | Weekly day containers | id (PK), dateStr, dayOfWeek, date, weekKey |
| `PlanTemplates` | Reusable plan templates | id (PK), name, description, activeDays (CSV) |
| `TemplateTasks` | Template task definitions | id (PK), templateId (FK→PlanTemplates), title, description, startTime, endTime, type, priority, energyLevel, estimatedCost |
| `Preferences` | Application key-value config | key (PK), value |
| `TodoItems` | Standalone todo entities | id (PK), title, description, completed, createdAt, itemType (note/timer/list), durationMinutes, checklistJson, audioFilePath |

### Migration Strategy
- **v1→v2:** Added sourceTemplateId FK, activeDays for recurring templates
- **v2→v3:** TodoItems table introduction
- **v3→v4:** Energy level & cost tracking fields
- **v4→v5:** Timer/audio/checklist support for TodoItems
- **Legacy Migration:** SharedPreferences JSON→Drift one-time migration with cleanup

### Model Classes
- **Task** - Immutable with `copyWith()` pattern, mutable `completed` field, JSON serialization, enum types (TaskType, TaskPriority, TaskEnergyLevel)
- **DayPlan** - Aggregates `List<Task>`, date mapping, JSON serialization
- **PlanTemplate** - Reusable templates with `List<Task>`, recurring schedule logic (`isRecurring` getter), `copyWith()` immutable updates
